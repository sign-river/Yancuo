# 考研错题记录软件设计草案

## 一、最核心的设计原则

### 1. Word 和 PDF 只能作为输出格式，不能作为核心数据库

软件内部不能直接把一个巨大的 Word 文档当作“错题库”。

因为 Word 文档不适合完成：

- 按标签筛选
- 批量修改优先级
- 查找重复题目
- 记录复习次数
- 保存每次修改历史
- 同时管理原图、题目、答案和公式
- 处理 AI 修改冲突

更合理的方式是：

- SQLite 数据库保存题目属性和关系
- 独立文件夹保存原始图片、裁剪图片和附件
- Markdown 与 LaTeX 保存可编辑的题目正文
- Word 和 PDF 在需要打印时临时生成

换句话说：

> 数据库是源文件，Word 和 PDF 是成品。

### 2. 原始图片永久保留

AI 识别可能出错，人工编辑也可能误删内容，所以每一道题都应保留：

- 原始图片
- 处理后的图片
- AI 识别结果
- 用户确认后的版本
- 历史修改记录

任何时候都可以返回原图重新识别，不能让 AI 第一次识别的结果覆盖原始资料。

### 3. AI 生成的内容不能直接静默覆盖

AI 批量处理完成后，应显示：

- 新增了哪些题
- 修改了哪些字段
- 删除了哪些内容
- 哪些公式识别不确定
- 哪些题疑似重复
- 修改前后差异

用户可以选择：

- 全部接受
- 接受选中项
- 拒绝选中项
- 手动编辑后接受
- 恢复旧版本

这相当于给错题库增加一个类似代码版本管理的“修改审查”机制。

# 二、建议的软件形态

第一版建议做成 Windows 桌面软件，而不是网页程序。

推荐技术方向：

- Python 作为主要开发语言
- PySide6 负责桌面界面
- SQLite 负责本地数据库
- SQLAlchemy 负责数据库模型和查询
- Pydantic Settings 负责配置读取和校验
- Markdown 与 LaTeX 负责题目内容
- Qt WebEngine 负责公式和题目预览
- python-docx 生成 Word
- HTML、CSS 和 WeasyPrint 生成 PDF

PySide6 是 Qt 官方提供的 Python 绑定，适合构建完整桌面界面；Qt WebEngine 可以在程序内渲染网页内容，因此适合显示 Markdown 和数学公式。

SQLAlchemy 可作为 SQLite 之上的数据访问层，避免整个程序到处直接拼接 SQL；Pydantic Settings 适合把配置文件、环境变量和敏感配置统一加载与校验。

Word 输出可以使用 python-docx 创建和更新`.docx`文件；PDF 则可以先生成 HTML 页面，再通过支持分页 CSS 的 WeasyPrint 输出。

# 三、软件总体架构

建议划分为六层。

## 1. 界面层

负责用户操作：

- 错题浏览
- 图片导入
- AI 任务创建
- 标签管理
- 搜索筛选
- 编辑题目
- 复习记录
- 打印排版
- 设置管理

## 2. 应用服务层

负责业务流程：

- 导入一批图片
- 创建错题
- 批量修改题目
- 合并重复题
- 生成复习计划
- 创建打印任务
- 导入外部修改
- 撤销 AI 修改

界面不能直接操作数据库，所有操作都经过应用服务层。

## 3. AI 服务层

负责：

- 图片识别
- 数学公式识别
- 题目结构化
- 错因分析
- 标签推荐
- 解法生成
- 知识点提取
- 重复题判断
- 批量修改
- 结果校验

AI 层应采用“提供商适配器”设计，不能把某一家 API 写死。

统一接口可以抽象为：

- 图像识别接口
- 文本生成接口
- 结构化输出接口
- 向量嵌入接口
- 文件批处理接口

以后可以自由切换不同服务商或本地模型。

## 4. 数据层

负责：

- SQLite 数据库
- 图片和附件目录
- 缓存目录
- 导出目录
- 日志目录
- 备份目录

## 5. 导入导出层

负责：

- 导入图片
- 导入 PDF
- 导入外部工作区
- 导入旧错题文档
- 导出 Word
- 导出 PDF
- 导出图片合集
- 导出可供 AI 修改的数据包
- 数据库完整备份

## 6. 插件层

第一版暂时不必开放给普通用户，但架构中应预留：

- 自定义 AI 提供商
- 自定义导出模板
- 自定义识别流程
- 自定义复习算法
- 自定义题目字段
- 自定义文件导入器

# 五、科目、章节和题型体系

不能把“高等数学、线性代数、概率论”的章节直接写死在代码里。

建议采用树状分类：

```text
考研数学
├── 高等数学
│   ├── 函数、极限与连续
│   ├── 一元函数微分学
│   ├── 一元函数积分学
│   └── 多元函数微积分
├── 线性代数
│   ├── 行列式
│   ├── 矩阵
│   ├── 向量组
│   └── 特征值与二次型
└── 概率论与数理统计
    ├── 随机事件与概率
    ├── 随机变量
    └── 参数估计
```

用户可以：

- 新增科目
- 修改章节名称
- 移动章节
- 合并章节
- 调整章节顺序
- 导入章节模板
- 导出章节模板

这样软件以后也能用于英语、政治、专业课，而不局限于数学。

# 六、标签系统

标签应和章节分类分开。

章节说明“这道题属于什么知识领域”，标签说明“它具有什么特征”。

例如一道题可以属于：

- 科目：高等数学
- 章节：一元函数积分学
- 题型：不定积分

同时拥有标签：

- 换元积分
- 分部积分
- 绝对值
- 定义域
- 二次做错
- 高频错误
- 需要背公式

## 标签功能

需要支持：

- 创建标签
- 删除标签
- 重命名标签
- 合并标签
- 设置标签颜色
- 设置父子标签
- 批量添加标签
- 批量删除标签
- 按标签筛选
- 标签交集筛选
- 标签并集筛选
- 排除某个标签
- 查看标签使用次数
- 查找没有标签的题目

## 系统标签和用户标签

建议分为两类。

系统标签由程序维护，例如：

- AI 待确认
- 疑似重复
- 缺少答案
- 缺少原图
- 识别失败
- 待复习
- 已掌握

用户标签由用户自由创建。

# 七、优先级系统

不建议只设置简单的“高、中、低”。

可以同时保存两个值：

## 1. 用户手动优先级

用户明确设定：

- 1：很低
- 2：较低
- 3：普通
- 4：重要
- 5：极重要

## 2. 系统计算优先级

系统根据以下因素计算：

- 重复做错次数
- 距离考试时间
- 知识点重要程度
- 最近复习结果
- 题目出现频率
- 用户手动优先级
- AI 判断的典型程度
- 是否属于薄弱章节

最终可以得到一个“建议复习分数”。

但系统计算结果不能覆盖用户手动优先级，只能作为排序参考。

# 八、图片导入流程

软件应支持四种输入方式：

## 1. 拖放图片

直接将一个或多个图片拖入软件。

## 2. 选择多个图片

通过文件选择器批量选择。

## 3. 选择文件夹

递归或非递归读取指定文件夹中的图片。

用户可以设置：

- 是否扫描子文件夹
- 图片扩展名
- 文件排序方法
- 是否跳过已经导入的图片
- 是否移动原文件
- 是否复制原文件
- 导入后是否自动创建 AI 任务

## 4. 监控文件夹

用户可以指定一个“收件箱文件夹”。

手机扫描或其他软件把图片传入这个文件夹后，程序自动发现新文件，并显示：

> 检测到 23 张新图片，是否导入？

第一版可以只做手动刷新，后续再增加实时监控。

# 九、AI 识别处理流程

建议不要让 AI 拿到图片后直接写入正式题库，而是经过以下阶段。

## 阶段一：文件预处理

程序完成：

- 图片旋转检测
- 方向修正
- 裁剪
- 透视矫正
- 去阴影
- 对比度增强
- 重复图片检测
- 图片哈希计算
- 图片质量检查

## 阶段二：页面识别

识别：

- 页面中有几道题
- 每道题的位置
- 题号
- 题目区域
- 用户作答区域
- 答案解析区域
- 手写标记
- 页码
- 章节标题

## 阶段三：题目结构化

AI 输出严格的结构化数据，而不是一大段自然语言。

例如：

```json
{
  "subject": "高等数学",
  "chapter": "一元函数积分学",
  "problem_number": "例 17",
  "question_markdown": "计算积分……",
  "question_latex": "\\int ...",
  "user_answer": "...",
  "correct_answer": "...",
  "knowledge_points": ["换元积分", "对数函数"],
  "error_types": ["绝对值遗漏"],
  "uncertain_fields": [
    {
      "field": "question_latex",
      "content": "ln x 或 ln|x|",
      "reason": "图片模糊"
    }
  ]
}
```

## 阶段四：程序校验

程序需要检查：

- JSON 结构是否正确
- 必填字段是否缺失
- 章节是否存在
- 标签是否合法
- 公式是否为空
- 图片引用是否存在
- 是否和已有题目重复
- AI 是否试图修改禁止修改的字段

## 阶段五：人工审核

界面采用左右对照：

- 左侧显示原图
- 中间显示 AI 识别文本
- 右侧显示公式渲染和分类结果

有问题的字段高亮显示。

## 阶段六：写入题库

只有用户接受后，才进入正式错题库。

# 十、外部编辑机制

这是整个软件很关键的一部分。

用户希望通过带 AI 的编辑器批量修改错题，因此软件需要提供一个稳定的“外部工作区格式”。

## 不建议直接让外部编辑器修改 SQLite 数据库

直接修改数据库存在很大风险：

- 字段损坏
- 关系丢失
- ID 改变
- 图片路径失效
- 多个程序同时修改
- 无法判断改了什么

## 推荐方案：工作区导出与同步

用户选择一批题目，点击：

> 导出到外部编辑工作区

软件生成：

```text
错题工作区/
├── manifest.json
├── instructions.md
├── problems/
│   ├── 题目 UUID-1/
│   │   ├── problem.md
│   │   ├── metadata.json
│   │   └── assets/
│   │       ├── original.jpg
│   │       └── answer.jpg
│   └── 题目 UUID-2/
│       ├── problem.md
│       ├── metadata.json
│       └── assets/
└── schemas/
    └── problem.schema.json
```

`problem.md`负责可读、可编辑的内容：

```text
---
id: 题目 UUID
revision: 7
subject: 高等数学
chapter: 一元函数积分学
priority: 5
tags:
  - 换元积分
  - 绝对值
---

# 原题

……

# 我的错误过程

……

# 正确解法

……

# 核心公式

……
```

用户可以让外部 AI 编辑器批量修改这些文件。

修改完成后，回到软件点击：

> 导入外部修改

程序完成：

- 检查文件格式
- 比较版本号
- 显示修改差异
- 检测字段冲突
- 验证图片路径
- 验证标签和章节
- 让用户选择接受哪些修改
- 写入新的历史版本

## 冲突处理

假设外部编辑期间，软件内部也修改了同一道题。

程序不能直接覆盖，而应显示：

- 数据库当前版本
- 外部文件版本
- 两边分别修改了哪些字段

用户可以：

- 保留内部版本
- 使用外部版本
- 按字段合并
- 创建题目副本

# 十一、软件内部 AI 编辑接口

软件中应提供一个“AI 操作台”。

用户可以选择：

- 当前题目
- 多个选中题目
- 当前筛选结果
- 某个章节
- 某个标签
- 一个图片文件夹
- 整个错题库

然后输入提示词。

例如：

- 为选中题目补充知识点
- 统一规范所有公式
- 查找可能遗漏绝对值的积分结果
- 将错因分为计算错误、概念错误和方法错误
- 删除重复的易错提醒
- 为每道题生成一道相似题
- 判断哪些题已经没有保留价值
- 给所有题补充简短解题路线

## AI 权限范围

每次执行前，用户需要选择 AI 可以修改的字段：

- 只读分析
- 允许修改标签
- 允许修改题目正文
- 允许修改答案
- 允许修改分类
- 允许创建新题
- 允许合并重复题
- 禁止删除题目

默认情况下，AI 不应拥有删除权限。

## 预设提示词

系统可以内置：

- 图片识别
- 题目结构化
- 错因分析
- 公式规范化
- 标签推荐
- 知识点提取
- 同类题归并
- 复习摘要
- 打印内容精简
- 相似题生成

用户也可以创建、修改和导入提示词模板。

提示词模板同样不能写死在代码中。

# 十四、重复题检测

重复题非常重要，否则错题库会逐渐膨胀。

可以分三层判断：

## 1. 图片完全重复

通过文件哈希或感知哈希判断。

## 2. 题目文本相似

比较识别后的题目文字。

## 3. 数学结构相似

由 AI 判断：

- 是否只是数字不同
- 是否使用相同核心方法
- 是否属于同一母题
- 是否值得分别保留

系统只提示“疑似重复”，不要自动删除。

用户可以：

- 保留两题
- 合并成一题
- 设置母题和变式题
- 只保留代表题
- 将另一题记录为重复错误次数

# 二十八、最关键的用户流程

## 流程一：批量录入

```text
选择图片或文件夹
→ 图片进入收件箱
→ 创建 AI 识别任务
→ AI 输出结构化结果
→ 用户检查不确定字段
→ 批量接受
→ 进入正式错题库
```

## 流程二：日常整理

```text
打开错题库
→ 按章节或标签筛选
→ 批量调整优先级
→ 补充错因和知识点
→ 标记需要复习
```

## 流程三：外部 AI 批量编辑

```text
选择一批题目
→ 导出工作区
→ 使用外部 AI 编辑器修改
→ 软件导入修改
→ 查看前后差异
→ 接受或拒绝
```

## 流程四：打印

```text
筛选需要打印的题目
→ 加入打印队列
→ 调整题目顺序
→ 选择输出内容
→ 预览自动排版
→ 导出 Word 或 PDF
```

## 流程五：复习

```text
打开今日复习
→ 隐藏答案重新做题
→ 记录结果和用时
→ 系统调整掌握度
→ 安排下次复习
```

# 二十九、这个软件真正的核心

真正困难的部分不是 AI 识别，也不是生成 Word。

最关键的是以下四件事：

1. 错题必须具有稳定的结构化数据。
2. 图片、正文、标签和复习记录不能互相混乱。
3. AI 修改必须经过验证、审查和版本管理。
4. 外部编辑器不能直接破坏内部数据库。

只要这四点设计正确，后面增加 AI 模型、复习算法、打印模板都会比较顺利。

反过来，如果一开始就以 Word 文档或一个大 JSON 文件作为中心，数据量变大以后，修改、筛选、同步和恢复都会非常痛苦。
