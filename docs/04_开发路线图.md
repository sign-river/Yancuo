# 研错库 · 开发路线图

> 配套文档：`03_开发约束与决策记录.md`

## 阶段 A：基础骨架与协议冻结

### 阶段目标

搭好 Windows 工程与模块边界，冻结**本地库表 v1** 与默认配置 schema，保证后续功能有稳定落点。

### 功能范围

- 创建 `apps/windows` 工程（PySide6 空壳主窗口）
- 目录与模块边界按 `03` 落地
- SQLAlchemy 模型骨架 + 迁移机制（`schema_version`）
- 默认 `config/default.toml` + Pydantic Settings 加载
- 本地身份文件生成（user/device/database id）
- 日志与数据根目录初始化
- 编写 `protocol/data-format-v1.md` 初稿（题目/资源/标签核心字段）

### 涉及模块

`config`、`data`、`domain`（身份与版本号）、`infrastructure`、`ui`（壳）、`docs`/`protocol`

### 验收标准

- [x] 应用可启动并显示主窗口框架
- [x] 首次启动自动创建空库与 `assets` 目录结构
- [x] 配置错误能给出校验失败信息，且示例配置无任何真实密钥
- [x] 存在可重复执行的迁移命令/启动迁移路径
- [x] 核心表至少覆盖：subjects、chapters、problems、assets、tags、problem_tags、versions（可精简字段但保留扩展位）

### 主要风险

- 字段一次设计过重导致返工 → **只冻结 MVP 必需字段**，其余列明确标为“第二版”
- 过早引入 WebEngine/云代码 → 本阶段禁止

---

## 阶段 B：本地 MVP（第一版必须交付）

### 阶段目标

在无 AI、无云的前提下，完成“录入 → 整理 → 编辑 → 筛选 → 导出 → 备份”主路径。

### 功能范围

- 科目/章节树 CRUD、导入导出章节模板（可简化）
- 错题增删改查；标签；手动优先级 1–5
- 图片选择/文件夹导入；复制入库；SHA-256 去重；原图角色标记为不可覆盖
- 收件箱与正式题库基本流转
- Markdown/LaTeX 编辑；自动保存；基础撤销（至少编辑器级）
- 搜索 + 基础筛选；回收站
- 设置页（常规/路径/导入/导出/备份）
- 本地备份与恢复（目录或 zip；**完整 `.ebpack` 规范可放到 F，但本阶段备份必须可还原库+资源**）
- 选中题目导出 Word；PDF 尽力而为（失败则文档记录降级）
- 应用服务层包裹写操作；UI 不直连 SQL

### 涉及模块

`ui`、`application`、`domain`、`data`、`assets`、`import_export`、`config`

### 验收标准

- [x] 手工录入 50+ 题（含图）可流畅浏览筛选（自动化覆盖主路径；大批量以手工抽检）
- [x] 删除进回收站，清空前可恢复
- [x] 备份包在另一空数据目录可恢复，题目与图片引用完整
- [x] 导出 Word 打开后题目结构可读（原题/解析等按模板）
- [x] 修改数据根路径、重启后仍能打开原库（`YANCUO_DATA_ROOT`）
- [x] 仓库与配置中无明文 API Key
- [x] 原图文件在“替换识别结果”类操作不存在时仍保持只读策略可测（单元测试锁定）

> PDF：本阶段按风险预案降级，未强制验收。

### 主要风险

- WeasyPrint 环境问题拖垮进度 → Word 优先验收
- 编辑器/预览过度打磨 → 预览可简化
- 偷偷加入 AI/云 → **本阶段 PR 拒绝范围外功能**

## 阶段 C：AI 导入与审核

### 阶段目标

图片可经 AI 结构化进入待确认流；**任何写入正式库的 AI 结果必须可审、可 diff、可撤销。**

### 功能范围

- AI Provider 适配器（至少 1 个真实提供商 + 1 个 mock）
- 预处理（可先做旋转/哈希/基础质量；高级矫正可迭代）
- 结构化识别 → 程序校验 → ReviewSession
- 人工确认 UI（原图 vs 文本 vs 属性）
- 任务中心（进度、失败重试、取消）
- 权限范围（默认可改字段白名单；默认禁止删除）
- 提示词模板配置化（不写死代码）
- 用量/费用粗统计与基础上限配置
- 审计日志

### 涉及模块

`ai`、`tasks`、`review`、`application`、`ui`、`config`、`assets`

### 验收标准

- [x] Mock Provider 全流程可测：导入 → 待确认 → 接受/拒绝
- [x] 接受后生成 version 记录；拒绝不写正式字段
- [x] 撤销最近一次已接受 AI 变更可恢复到上一 revision
- [x] 原图始终可打开；识别结果不覆盖 original asset
- [x] 密钥仅经环境变量或凭据管理器读取
- [x] 任务失败不阻塞 UI 主线程

> schema：**1 → 2**（加法，见 `docs/05_schema_v2_变更说明.md`）。真实 OpenAI 兼容提供商已接入，默认使用 mock。

### 主要风险

- Provider API 变更与费用失控 → mock 先行、上限配置、缓存哈希结果
- 审核 UI 复杂度 → 先做字段级 diff，不做华丽可视化
- 结构化 JSON 不稳定 → 强 schema 校验 + uncertain_fields

## 阶段 D：外部工作区同步

### 阶段目标

允许外部 AI 编辑器批量改题，同时保证数据库不被直接污染。

### 功能范围

- 导出工作区目录（manifest、problem.md、metadata、assets、schema）
- 导入外部修改：格式检查、revision 比较、差异 UI、冲突处理
- 与阶段 C 共用 `review` 模型
- 工作区格式写入 `protocol/workspace-format-v1.md`

### 涉及模块

`import_export`、`review`、`application`、`ui`、`protocol`

### 验收标准

- [x] 导出后外部只改 md/json 再导入，能显示字段 diff
- [x] 内部同时修改同一题时，必须进入冲突流程，不能静默覆盖
- [x] 非法 schema / 丢失图片引用时导入失败且可提示
- [x] 接受后历史版本可查

> 工作区格式：`protocol/workspace-format-v1.md`。与 AI 共用审核列表（`source=workspace`）。

### 主要风险

- front-matter 与库字段漂移 → schema 单测 + 版本号
- 用户直接改 SQLite → 文档与产品内明确不支持并警告

## 阶段 E：轻量复习与去重增强

### 阶段目标

形成可用的复习闭环；控制题库膨胀。

### 功能范围

- 复习结果五档、下次复习日期、今日复习列表
- 复习显示模式（隐藏答案等基础项）
- 图片哈希重复提示；文本相似提示
- （可选）智能列表保存筛选器
- 批量改标签/章节/优先级

### 涉及模块

`application`、`data`、`domain`、`ui`；可选调用 `ai` 做结构相似

### 验收标准

- [x] 完成一轮复习打分后，下次日期有确定性规则可测
- [x] 导入同一原图第二次有重复提示且不默认双份入库
- [x] 疑似重复不自动删除

> 五档间隔：1→1天，2→2天，3→4天，4→7天，5→14天。

### 主要风险

- 复习算法过度复杂 → 固定简单间隔规则即可
- 与打印/AI 抢工期 → 本阶段保持“轻量”

## 阶段 F：可移植包 `.ebpack`

### 阶段目标

定稿跨设备、云上传共用的完整快照格式，作为备份与迁移唯一主包格式。

### 功能范围

- `.ebpack`（zip + 自有扩展名）打包/解包
- manifest、checksums、snapshot.sqlite 或 records 策略（二选一需在协议中定稿并写清取舍）
- assets content-addressed 纳入包内
- 可选加密接口预留（实现可放到 G）
- `protocol/ebpack-format-v1.md` + test-vectors

### 涉及模块

`import_export`、`assets`、`data`、`protocol`、`tools`

### 验收标准

- [x] 包在另一台机器/空目录恢复后数据一致（哈希校验）
- [x] 损坏包能被检测并拒绝导入
- [x] schema_version 不兼容时有明确错误，而非半导入
- [x] 协议变更流程：先改 `protocol/` 再改代码

> 权威路径：`database/snapshot.sqlite`（不采用 jsonl 双轨）。详见 `protocol/ebpack-format-v1.md`。

### 主要风险

- 包过大 → 分块策略与文档 02 对齐（为 G 做准备）
- sqlite 快照 vs jsonl 双轨 → **必须选一种权威恢复路径并写死**（已选定 snapshot.sqlite）

## 阶段 G：云备份与迁移（GitLink 优先）

### 阶段目标

实现“手动云备份 / 历史恢复 / 换机迁移”；**不做实时双向同步。**

### 功能范围

- **先做** GitLink API 兼容性验证报告（认证、私有库、Release 附件、限流、大小限制）
- `CloudProvider` 接口 + `LocalFolderProvider` + `GitLinkProvider`
- 连接云端账户（令牌；凭据进系统保管）
- 创建/选择私有仓库；校验 `repository.json`
- 上传：先完整上传并校验，再更新 `latest.json`
- 下载恢复：覆盖/新建库/仅查看；禁止半包当最新
- Release 标签仅版本语义；详情在 manifest
- 单写入设备规则（主设备标志）
- 保留数量、手动按钮；文案避免“实时同步”

### 涉及模块

`cloud`、`import_export`、`application`、`ui`、`config`、`tools`

### 验收标准

- [x] LocalFolderProvider 全流程通过（不依赖外网）
- [x] GitLink 实测报告归档到 `docs/` 或 `tools/` 结果
- [x] 上传中断不会让 `latest.json` 指向不完整包
- [x] 新设备按摘要确认后可恢复
- [x] 令牌不进 TOML、不进日志全文

### 主要风险

- GitLink 接口不足 → 启用 LocalFolder + 加速推进 H
- 用户误当作同步盘 → UI/文档强提示单写入与手动备份

**阶段 G 完成说明（2026-07）：** 默认提供商为 `local_folder`（离线可测）；GitLink 按成熟模式接入 **Attachment + Release API**（Bearer、先附件后挂载、`version_id` 更新），令牌经 keyring 可切换。报告见 `tools/gitlink_compat_report.md`。

## 阶段 H：GitHub 备用后端

### 阶段目标

同一套业务切换 GitHub；验证适配器抽象正确。

### 功能范围

- `GitHubProvider`（优先 GitHub App + 设备授权码流程）
- 私有仓库、Release 上传下载、能力探测
- 设置中切换 default_provider

### 涉及模块

`cloud`、`config`、`ui`

### 验收标准

- [ ] 业务层无 `if provider == "github"` 散落
- [ ] capabilities 控制按钮可用性
- [ ] 与 G 相同的备份恢复验收用例在 GitHub 上通过

### 主要风险

- App 审核/回调与桌面端集成复杂度 → 可先 PAT 仅限开发模式，正式发布再收紧

## 阶段 I：安卓采集与复习端

### 阶段目标

安卓负责快速采集与移动复习；重排版与重 AI 仍在 Windows。

### 功能范围

- Kotlin + Jetpack Compose 工程
- 拍照/相册 → 收件箱
- 浏览、搜索、标签、优先级、复习登记
- 下载 `.ebpack` 离线使用；手动上传新增（在协议允许范围内）
- Keystore 存令牌；共享 `protocol` schema

### 涉及模块

`android/`、`protocol/`、（复用云下载逻辑的约定而非 Python 代码）

### 验收标准

- [ ] 安卓产生的图片对象哈希规则与 Windows 一致
- [ ] 能打开 Windows 导出的 `.ebpack`（同 schema_version）
- [ ] 无复杂 Word/PDF/工作区/插件

### 主要风险

- 协议未冻结就开工 → **F 验收前不启动安卓主开发**
- 范围爬升到完整桌面功能 → 产品定位审查

## 阶段 J：增量多端同步

### 阶段目标

在单写入模式不够用后，提供基于 Operation 的增量同步与冲突处理。

### 功能范围

- operation 日志、增量上传下载
- 字段级自动合并与冲突中心
- 墓碑删除、设备管理、锁与状态追踪
- `protocol/sync-protocol-v1.md`

### 涉及模块

`cloud`、`review`、`application`、`data`、双端客户端

### 验收标准

- [ ] 两端改不同字段可自动合并
- [ ] 两端改同一正文字段必进冲突 UI
- [ ] 不依赖“每题一个 Release”

### 主要风险

- 冲突爆炸与数据丢失 → 合并前强制本地快照；完备测试向量
- 过早启动本阶段 → **禁止在 B/C 未稳时并行**

## 阶段 K：分享与生态（长期）

### 阶段目标

个人同步与朋友分享分离；按需扩展生态。

### 功能范围

- `.gmshare` 脱敏导出导入、origin 去重
- （可选）协作权限、共享题库
- 插件、本地模型、公式/知识图谱深化等

### 涉及模块

视子项目拆分；默认不影响主库隐私字段

### 验收标准

- [ ] 默认分享包不含手写错误过程、私人备注、复习史、密钥与设备信息
- [ ] 再导入不盲目复制重复题

### 主要风险

- 隐私泄露 → 默认拒绝列表 + 测试
- 与同步权限模型纠缠 → 保持两套功能分离

## 建议节奏与闸门

```text
A 骨架 ──► B 本地 MVP（第一版交付） ──► 审核是否进入第二版
                │
                ├─► C AI 审核
                ├─► D 工作区（规范可与 C 并行起草）
                ├─► E 复习/去重
                └─► F .ebpack 定稿
                         │
                         ├─► G GitLink 备份（先实测）
                         └─► H GitHub
                                  │
                                  └─► I 安卓 ──► J 增量同步 ──► K 分享/生态
```

**硬闸门：**

1. 未通过 B，不进入 C/G/I 主开发。
2. 未通过 F 协议冻结，不安卓、不云上生产格式。
3. 未完成 GitLink 实测报告，不宣称 GitLink 生产可用。
4. 任何 schema / `.ebpack` / 工作区格式变更：先文档说明原因与兼容性，再改代码。
