> 本文档基于 `01_产品功能与本地架构.md` 与 `02_云端同步与安卓端设计.md` 整理。

## 1. 核心目标、目标用户与关键流程

### 1.1 核心目标

研错库是面向考研备考的**本地优先错题管理系统**。真正要解决的不是“把 AI 接上”或“生成 Word”，而是：

1. 错题具备稳定、可查询、可版本化的结构化数据。
2. 原图、正文、标签、复习记录彼此解耦且不互相污染。
3. AI / 外部编辑的修改必须经过校验、差异审查与可撤销写入。
4. 外部工具不能直接破坏本地 SQLite 工作库。

一句话：**SQLite + 对象存储图片是源；Word / PDF / 工作区 / 云端包只是产物或传输形态。**

### 1.2 目标用户

| 角色                     | 特征                                           | 主要诉求                               |
| ------------------------ | ---------------------------------------------- | -------------------------------------- |
| 考研学生（主用户）       | 数学等高密度公式科目，题目多、重复错、需要打印 | 快速录入、整理分类、复习、导出打印     |
| 使用 AI 编辑器的进阶用户 | 希望批量规范化公式、补知识点、归并重复题       | 安全的外部工作区导入导出，而非直接改库 |
| 多设备用户（后续）       | Windows 整理 + 安卓采集/复习                   | 可靠备份迁移，再谈增量同步             |
| 朋友间分享（长期）       | 交换精选题，不想暴露私人笔记与复习史           | 脱敏分享包，而非共享账号协作           |

### 1.3 最重要的使用流程（按价值排序）

1. **批量录入**：图片/文件夹 → 收件箱 →（可选 AI）结构化 → 人工确认 → 正式题库
2. **日常整理**：筛选 → 改标签/优先级/章节 → 补错因与知识点
3. **外部 AI 批量编辑**：导出工作区 → 外部修改 → 差异审查 → 接受/拒绝
4. **打印导出**：筛选入队 → 排版 → Word / PDF
5. **复习**（轻量即可）：今日复习 → 隐藏答案重做 → 记录结果 → 调整掌握度
6. **备份与迁移**（云端第一阶段）：完整 `.ebpack` → Release 快照 → 新设备恢复

## 2. 两份设计的矛盾、重复、缺失与实现风险

### 2.1 矛盾 / 需统一的决策

| 议题                   | 文档 01                                             | 文档 02                                                    | 决议                                                                                                          |
| ---------------------- | --------------------------------------------------- | ---------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------- |
| 开发阶段划分           | 本地 → AI → 工作区 → 复习智能化 → 高级（含云/手机） | 本地+`.ebpack` → GitLink → GitHub → 安卓 → 增量同步 → 分享 | **以本地稳定为先**；`.ebpack` 随本地备份一起落地；云与安卓整体后移。采用本文档第 3 节与 `04` 路线图统一阶段。 |
| 第一版是否含 AI        | 明确“第一阶段先不接 AI”                             | 本地核心版未强制 AI，但也未强调延后                        | **第一版（本地 MVP）不接 AI**；AI 作为紧随其后的独立阶段。                                                    |
| 第一版是否含 `.ebpack` | 有“完整备份”，未定义 `.ebpack`                      | 本地核心版即含 `.ebpack`                                   | **本地 MVP 先做目录式完整备份**；协议化 `.ebpack` 在“可移植包”阶段定稿并实现。                                |
| 配置示例路径           | `./data/error_book.db` 等                           | 增加 `[cloud]` / `[sync]` / `[encryption]`                 | **统一 TOML**；云相关配置可先预留 schema，第一版不启用逻辑。                                                  |
| 产品命名               | 草案未统一命名                                      | GraduateMistakeBook / 研错库                               | **对外中文名：研错库；仓库/包标识：`graduate-mistake-book`；工程名 Yancuo。**                                 |
| 安卓技术               | 文档 01 把手机端放在第五阶段且偏笼统                | Kotlin + Compose 优先，不共享 Python UI                    | **采纳文档 02**：共享协议，不共享 UI/语言。                                                                   |

### 2.2 重复（应合并为单一规范，避免双份真相）

- 图片去重 / 相对路径 / 禁止绝对路径：两份均强调 → 写入 `protocol` 与数据层唯一实现。
- 版本审查与差异接受：AI 审核、外部工作区导入、云端合并三处重复 → 统一为 **ChangeSet / ReviewSession** 模型。
- 配置与密钥：TOML + 凭据管理器 → 单一 `config` 模块。
- 备份：本地备份、`.ebpack`、Release 快照 → 分层说明，避免三套互不兼容的打包逻辑。

### 2.3 缺失（实现前必须补齐）

1. **权威 schema**：题目、标签、版本、AI Job、操作事件的字段级定义与 JSON Schema 尚未定稿。
2. **数据库迁移策略**：`schema_version`、向前兼容规则、失败回滚未写清。
3. **GitLink API 实测定稿**：认证、Release 附件、限流、大文件限制必须先做兼容性验证，不能推测实现。
4. **工作区格式版本**：`manifest.json` / `problem.md` front-matter / revision 语义未形成正式规范。
5. **收件箱与正式题库状态机**：题目状态枚举、何时可打印/可同步未统一。
6. **加密细节**：口令派生算法、AEAD、恢复密钥 `.gmkey` 格式、与 Release 的关系未定。
7. **单写入设备锁**：锁超时、主设备转让、离线持锁失败策略未定义。
8. **错误码与日志规范**：任务失败、导入冲突、云上传失败的用户可见语义缺失。
9. **测试向量**：跨端哈希、加密、包校验缺少 `protocol/test-vectors`。
10. **许可证与第三方依赖边界**（WeasyPrint、WebEngine 等）未评估。

### 2.4 实现风险（高优先级）

| 风险               | 说明                                     | 缓解                                                   |
| ------------------ | ---------------------------------------- | ------------------------------------------------------ |
| 范围膨胀           | 两份文档功能面极广，易同时做 AI/云/安卓  | 严格按路线图闸门；未审核不扩需求                       |
| AI 静默写库        | 识别结果直接覆盖原图或正式字段           | 原图只读；写入必须经 Review；默认禁止删除              |
| 以文档当库         | 用户习惯导出 Word 后当主库改             | 产品文案与导入路径强调“库为源”                         |
| 云当成实时同步     | Release 被用于每题上传                   | 第一版云仅备份迁移；UI 文案禁止“实时同步”表述          |
| GitLink 能力不确定 | 附件/鉴权与文档假设不符                  | 独立“兼容性验证”阶段，失败则 LocalFolder + GitHub 兜底 |
| 路径与编码         | Windows 中文路径、长路径、图片名         | 全程 content-addressed 对象库 + 相对路径               |
| PDF 依赖脆弱       | WeasyPrint 原生依赖在 Windows 安装成本高 | MVP 可先稳 Word；PDF 作为可降级能力单独验收            |
| 公式预览重量       | Qt WebEngine 体积与内存                  | 预览可后置；编辑器先纯文本 + 延迟渲染                  |
| 双端协议漂移       | Windows / 安卓各自改字段                 | `protocol/` 为唯一真相；改格式先改规范再改代码         |
| 密钥进仓           | 配置或示例写入真实 Key                   | 仅 `*_env` / `credential_key`；CI 扫描敏感信息         |

## 3. 需求划分

### 3.1 第一版必须实现（Windows 本地 MVP）

目标：**录入、整理、筛选、编辑、打印（或至少 Word 导出）、本地备份** 稳定可用；不接 AI、不做云、不做安卓。

- 本地身份：`user_id` / `device_id` / `database_id`（不依赖云）
- SQLite 工作库 + 图片对象库（SHA-256 相对路径）
- 科目 / 章节树 / 题型（可配置，不写死考研数学）
- 错题 CRUD、标签、手动优先级
- 图片导入（选择文件/文件夹；复制入库；跳过重复哈希）
- 收件箱 → 正式题库的基本状态流转（可先手工整理，不经 AI）
- Markdown + LaTeX 题目编辑（预览可简化）
- 搜索与基础筛选（科目、章节、标签、优先级、时间）
- 配置 TOML + 设置界面 + Pydantic 校验
- 回收站、自动/手动本地备份、完整性检查（库+资源引用）
- 选中题目导出 **Word**（PDF 力争同期，允许降级为“已知限制”）
- 操作不直连 SQL：经应用服务层

**明确不做：** AI、外部工作区、云 Release、安卓、增量同步、朋友分享、插件市场、复杂复习算法、公式/知识点完整知识图谱。

### 3.2 第二版实现

在 MVP 稳定后按依赖顺序推进（详见路线图阶段，不必同一发布号一次做完）：

- AI 提供商适配器、识别流水线、任务中心、人工审核与差异接受
- 外部编辑工作区导出/导入、revision 冲突处理、历史版本
- 轻量复习（结果登记、下次复习日期、今日复习）
- 重复题检测（哈希 + 文本相似；AI 结构相似可并行或稍后）
- 可移植包 `.ebpack` 定稿（备份/迁移载体）
- 云端适配器骨架 + `LocalFolderProvider`
- GitLink 兼容性验证 → 手动备份/恢复（单写入设备）
- GitHub 备用后端（App / 设备码流程）
- 导出模板基础化、智能列表、批量操作增强
- 成本控制与 AI 审计日志基础项

### 3.3 长期规划

- 安卓端（Kotlin/Compose）：采集、收件箱、浏览、标签/优先级、复习、快照下载与手动上传
- 真正的多设备增量同步、字段级合并、冲突中心、墓碑删除
- 端到端加密与恢复密钥体系完善
- 朋友分享包 `.gmshare`、协作/共享题库
- 公式库 / 知识点库深化、智能优先级、相似题生成
- 插件机制、本地模型、浏览器扩展
- 多用户模式、在线社区等（非核心，默认更靠后）

### 4.2 Windows 应用内部模块边界

```text
apps/windows/src/yancuo_win/
├── main.py
├── ui/                    # 界面层：只调 application，不碰 SQL/文件系统细节
├── application/           # 应用服务：用例编排、事务边界、撤销入口
├── domain/                # 领域模型与纯规则（题目状态、权限、合并规则）
├── data/                  # SQLAlchemy 模型、仓储、迁移
├── assets/                # 图片对象库（哈希、去重、只读原图策略）
├── import_export/         # 图片导入、Word/PDF、备份包、（后）工作区/ebpack
├── ai/                    # AI 适配器与流水线（第二版启用）
├── cloud/                 # CloudProvider 适配器（第二版后半启用）
├── review/                # ChangeSet 审核、差异、接受/拒绝（AI/工作区/同步共用）
├── tasks/                 # 后台任务队列与任务中心模型
├── config/                # Pydantic Settings、TOML、凭据解析
└── infrastructure/        # 日志、路径、OS 凭据、进程单例等
```

**边界规则：**

- `ui` → 仅依赖 `application`
- `application` → 可依赖 `domain` / `data` / `assets` / `import_export` / `review` / `tasks`
- `ai` / `cloud` → 不得被 `ui` 直接调用；经 `application`
- 禁止业务代码散落 `if provider == "github"`；平台差异留在适配器
- 配置常量禁止写死在 `application`/`domain`；读 `config`

## 5. Windows 端第一版技术选型

| 选型         | 选择                                   | 理由                                                   |
| ------------ | -------------------------------------- | ------------------------------------------------------ |
| 语言         | Python 3.11+                           | 与设计一致；生态覆盖桌面、ORM、文档导出、后续 AI SDK   |
| UI           | PySide6                                | 官方 Qt 绑定，适合桌面复杂布局（三栏列表+编辑器）      |
| 本地库       | SQLite                                 | 离线优先、单文件易备份、满足结构化查询                 |
| ORM          | SQLAlchemy 2.x                         | 避免拼接 SQL；迁移与模型清晰                           |
| 配置         | TOML + Pydantic Settings               | 可读、可校验；支持环境变量与密钥引用                   |
| 题目正文     | Markdown + LaTeX                       | 可编辑、可 diff、适合外部工作区                        |
| 公式预览     | MVP 可简化；完整版 Qt WebEngine        | WebEngine 重，MVP 可先延迟或用轻量方案，避免阻塞主路径 |
| Word         | python-docx                            | 设计指定；打印主路径之一                               |
| PDF          | HTML/CSS + WeasyPrint（可降级）        | 分页 CSS 友好；Windows 依赖风险高，验收允许“Word 优先” |
| 打包         | 后期再定（如 PyInstaller / briefcase） | MVP 以源码/开发运行验收，避免过早纠缠打包              |
| 测试         | pytest                                 | 领域规则、导入导出、迁移优先单测                       |
| 凭据         | Windows Credential Manager             | API Key / 令牌不进 TOML、不进仓库                      |
| 云（非 MVP） | 统一 `CloudProvider`                   | GitLink / GitHub / LocalFolder 可切换                  |

**刻意不选（第一版）：** 纯 Web 前端当主程序、以 Word/JSON 大文件当库、Android Python 框架、把 Release 当消息队列。

## 6. 需优先确定的数据结构、配置格式与接口协议

下列项应在对应阶段**编码前**写成 `protocol/` 或 `docs` 附录并评审；变更须说明兼容性。

### 6.1 数据结构（优先）

1. **Problem 核心字段与状态机**
   - 状态：`inbox` / `active` / `archived` / `trashed` 等
   - `revision`、禁止 AI 修改字段列表
2. **Asset 对象模型**
   - `sha256`、mime、尺寸、角色（original / processed / answer / user_work）
   - 原图不可变策略
3. **Tag / Chapter / Subject**
   - 树形章节、系统标签 vs 用户标签
4. **Version / ChangeSet / ReviewSession**
   - 统一支撑 AI 审核、工作区导入、（未来）同步合并
5. **AI Job / JobItem**（第二版）
6. **Operation 事件**（增量同步长期）
7. **本地身份 Identity**（`user_id` / `device_id` / `database_id`）
8. **`.ebpack` manifest**（schema_version、checksums、加密标志）
9. **工作区 `manifest.json` + `problem.md` front-matter**
10. **云端 `repository.json` / `latest.json` / Release 命名**

### 6.2 配置格式（优先）

- 主配置：`config.toml`（或用户目录下等价文件），章节至少覆盖文档 01 的 `[application]/[paths]/[import]/[ai]/[export]/[backup]/[privacy]`
- 云相关：`[cloud]` / `[sync]` / `[encryption]` **可先定义默认值，MVP 逻辑关闭**
- 密钥：仅 `api_key_env` / `credential_key`，禁止明文
- 配置 schema 版本号与“旧配置备份”策略

### 6.3 接口协议（优先）

| 协议                                   | 用途                               | 何时定稿                     |
| -------------------------------------- | ---------------------------------- | ---------------------------- |
| 内部 Application API（服务层方法边界） | UI 与用例解耦                      | MVP 开工前                   |
| AI Provider 适配器接口                 | vision / text / structured / embed | AI 阶段前                    |
| 外部工作区格式 v1                      | 导出导入与冲突                     | 工作区阶段前                 |
| `.ebpack` v1                           | 备份迁移                           | 可移植包阶段前               |
| `CloudProvider` 接口 + capabilities    | GitLink/GitHub/Folder              | 云阶段前                     |
| Release 标签命名 + manifest/preview    | 快照识别                           | 云阶段前                     |
| 同步 Operation / 冲突规则              | 增量同步                           | 长期阶段前；不可提前绑架 MVP |
| 安卓共享 Schema                        | 跨端                               | 安卓阶段前；字段变更冻结窗口 |

## 7. 强制开发原则（后续实现必须遵守）

1. **SQLite 是本地工作数据库。**
2. **Word 和 PDF 只是导出格式，不是主数据载体。**
3. **原始图片不得被 AI 识别结果覆盖。**
4. **AI 修改必须支持审核、差异比较和撤销。**
5. **配置项不能写死在业务代码中。**
6. **API 密钥不得明文写入仓库。**
7. **GitLink 和 GitHub 必须通过统一云端适配器接入。**
8. **Release 主要用于完整快照、备份和迁移，不承担高频实时同步。**
9. **Windows 端和安卓端共享数据协议，但不强求共享 UI 代码。**
10. **第一阶段优先保证本地功能稳定，不提前实现复杂多端同步。**
11. **每次准备修改架构或数据格式前，先说明原因及兼容性影响。**

### 7.1 已确认的架构决策（ADR 摘要）
