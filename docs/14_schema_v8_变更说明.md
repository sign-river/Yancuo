# schema v8 变更说明

## 背景

笔记与错题需要共享分类和标签，但其内容结构、生命周期及后续 AI 提取流程不同。schema v8 先建立独立且最小的笔记权威模型，不把笔记内容、原图或块编辑状态塞入 `Problem`。

## 迁移

| 项目 | 变更 |
| --- | --- |
| `schema_version` | **7 → 8** |
| `note_documents` | 新增笔记文档：标题、摘要、科目、章节、状态、版本与删除时间 |
| `note_blocks` | 新增有序块：标题、文本、概念、公式、图片、提示；保留 Markdown、LaTeX、来源区域与 AI 不确定字段 |
| `note_assets` | 新增笔记原图/派生资产元数据；文件存储策略沿用本地对象目录 |
| `note_tags` | 新增笔记与既有 `tags` 的多对多关联 |
| `data_format_version` | 保持 **1** |

`NoteDocument` 可使用现有 `subjects` 与 `chapters` 分类；`NoteBlock` 通过 `sort_order` 定义文档内顺序。`NoteAsset` 只保存资源元数据，不替代原始文件。

## 生命周期与数据边界

- 笔记状态沿用 `inbox`、`active`、`archived`、`trashed`；回收站内容不可编辑，恢复后才可继续改动；
- 永久删除只允许回收站笔记，并会删除关联的块、资产元数据与标签关系；
- 题目答案、作答、复习次数、错因和题目详情字段不会出现在笔记表；
- 当前 `SearchDocument` 和 FTS5 仍只投影题目。`SEARCH-07` 会在笔记阅读/编辑面稳定后设计统一检索投影与普通/AI 搜索边界；
- `.ebpack` 完整 SQLite 快照会保留新增表，但跨端字段协议仍为 v1，Android 尚未提供笔记 UI 或笔记业务同步承诺。

## 迁移保护

沿用 schema v7 的保护流程：已有数据库升级前以 SQLite online backup API 生成副本、校验完整性与旧版本；迁移或核心表验证失败时关闭连接池并原子恢复该副本。新建空库不生成无意义备份。

## 后续范围

本次 schema 迁移完成 `NOTE-01` 数据基础；`NOTE-02` 已在 Windows 端提供笔记首页、阅读预览与基础块编辑器；`NOTE-03` 已接入图片视觉 Provider、可编辑草稿和确认入库；`NOTE-04` 已复用统一离线公式渲染、持久展示来源区域并提供原图按需查看。上述增强均复用 schema v8 预留字段，没有追加迁移。分类组确认与可选题目关联（`NOTE-05`）及统一搜索（`SEARCH-07`）仍未交付。
