# 研错库云端账户、同步与安卓端设计修订案

## 一、总体架构修正

软件应形成三层数据结构：

### 第一层：本地工作数据库

每台设备都有独立的本地数据：

- SQLite 数据库
- 原始题目图片
- 裁剪图片
- 用户作答图片
- 缓存
- AI 原始响应
- 导出文件
- 操作日志

日常增删改查全部操作本地数据库。

即使没有网络，用户仍然可以：

- 查看题目
- 编辑错题
- 添加标签
- 进行复习
- 导出 PDF 和 Word
- 导入图片

### 第二层：可移植数据包

软件将本地数据转换成统一的数据包格式，例如：

```text
user-data-20260721-150930.ebpack
```

`.ebpack`本质可以是一个 ZIP 压缩包，但使用软件自己的扩展名。

其中包括：

```text
package/
├── manifest.json
├── database/
│   ├── snapshot.sqlite
│   └── migrations.json
├── records/
│   ├── problems.jsonl
│   ├── tags.jsonl
│   ├── reviews.jsonl
│   └── formulas.jsonl
├── assets/
│   ├── objects/
│   │   ├── ab/
│   │   │   └── abcdef...jpg
│   │   └── cd/
│   │       └── cdef12...png
│   └── index.json
├── settings/
│   └── portable-settings.toml
├── checksums.sha256
└── package-signature.json
```

它具有四个用途：

1. 本地完整备份
2. 上传至 GitLink 或 GitHub
3. 在另一台 Windows 设备恢复
4. 在安卓端读取和同步

### 第三层：远程版本仓库

远程 GitLink 或 GitHub 仓库只负责：

- 保存同步清单
- 识别最新版本
- 保存设备信息
- 保存数据包索引
- 保存备份版本
- 下载和恢复数据
- 必要时保存增量同步记录

它不直接参与软件日常查询。

# 二、不要把所有识别字段写在 Release 标签中

你提到“通过创建 Release 的标签等用于读取辨认字段”，这个思路需要调整。

Release 标签适合表示：

- 数据格式版本
- 快照序号
- 创建时间
- 稳定性状态

例如：

```text
data-v1.4.0-snapshot-20260721T071530Z
```

标签中不应保存：

- 用户名
- 科目数量
- 数据库路径
- 题目数量
- 设备 ID
- 文件列表
- 图片分块信息
- 加密参数
- 数据库版本详情

这些信息应当放在 Release 附件中的`manifest.json`。

原因是标签：

- 不适合复杂结构化信息
- 长度和字符使用容易出问题
- 后期增加字段非常困难
- 不能很好地进行完整性校验
- 重命名后可能影响版本识别

## 推荐的 Release 结构

```text
Release
├── 标签：
│   data-v1.4.0-snapshot-20260721T071530Z
│
├── 名称：
│   研错库数据备份 · 2026-07-21 15:15
│
└── 附件：
    ├── manifest.json
    ├── snapshot.ebpack
    ├── snapshot.ebpack.sha256
    └── preview.json
```

`preview.json`仅保存少量可公开展示的非敏感信息：

```json
{
  "application": "GraduateMistakeBook",
  "schema_version": 4,
  "created_at": "2026-07-21T15:15:30+08:00",
  "problem_count": 826,
  "asset_count": 1427,
  "encrypted": true
}
```

GitHub Release 目前支持每个 Release 最多关联 1000 个附件，每个附件需小于 2 GiB。虽然限制看起来比较宽松，软件仍不应把数据包做得接近上限，建议每个上传分块控制在 100 至 500 MB，以降低上传失败和重试成本。

# 三、Release 的正确定位

Release 适合用于：

- 完整备份
- 手动数据迁移
- 某个稳定时间点的快照
- 灾难恢复
- 历史版本回退

Release 不适合用于：

- 每修改一道题就发布一次
- 实时多设备同步
- 高频增量上传
- 多设备同时写入
- 保存每张图片的每次变化

否则用户做十道题，就可能生成大量 Release，远程仓库很快变得难以管理。

## 推荐分工

### Git 分支负责轻量同步信息

例如专门建立：

```text
sync
```

分支，保存：

```text
sync/
├── remote.json
├── latest.json
├── devices.json
├── locks/
├── changes/
└── manifests/
```

### Release 负责完整快照

例如：

- 每天最多自动生成一次
- 用户点击“立即云备份”时生成
- 大规模 AI 修改前生成
- 数据库迁移前生成
- 新设备首次同步前生成

# 四、账号体系需要拆成两种账号

## 1. 软件本地身份

用户第一次启动时，软件创建：

- 用户 UUID
- 用户显示名称
- 当前设备 UUID
- 数据库 UUID
- 首次创建时间
- 本地加密密钥信息

例如：

```json
{
  "user_id": "usr_2a938...",
  "device_id": "dev_win_01f8...",
  "database_id": "db_6703...",
  "display_name": "张三"
}
```

这个身份不依赖 GitLink 或 GitHub。

因此，即使用户没有连接任何远程平台，也可以正常使用软件。

## 2. 云端提供商身份

用户可以连接：

- GitLink
- GitHub
- 将来可能增加的 Gitee
- 自建 Gitea
- WebDAV
- 对象存储

界面中的用词建议是：

> 连接云端账户

而不是简单写：

> 登录研错库

否则用户会误以为 GitLink 账号就是软件账号。

# 五、登录和授权方式

## GitHub

正式发布给朋友使用时，优先考虑 GitHub App，而不是让每个人手动复制一个权限很大的个人访问令牌。

GitHub 官方建议一般优先使用 GitHub App，因为它可以提供更细粒度的仓库权限、让用户选择授权哪些仓库，并可使用短期令牌。桌面应用还可以使用设备授权流程，让用户在浏览器输入一次性验证码完成连接。

推荐流程：

```text
点击“连接 GitHub”
→ 软件显示一次性验证码
→ 浏览器打开 GitHub 授权页面
→ 用户确认授权
→ 软件获得受限令牌
→ 用户选择或创建数据仓库
```

软件不能：

- 获取用户 GitHub 密码
- 在配置文件中明文保存访问令牌
- 默认申请整个账户的所有仓库权限
- 把访问令牌写入日志

令牌应保存在：

- Windows Credential Manager
- Android Keystore

## GitLink

GitLink 连接层需要允许多种认证模式：

- 平台官方授权流程
- API 令牌
- Git 凭据
- 用户手动配置令牌

实际开发前，应专门验证 GitLink 正式环境中的：

- 令牌创建方式
- 私有仓库访问
- 创建 Release
- 上传 Release 附件
- 下载附件
- 删除 Release
- API 限流
- 大文件限制
- 令牌过期机制

当前 GitLink 公开 API 文档能够确认项目、仓库、分支、标签、文件树等能力，也支持创建私有项目；但是认证和 Release 附件接口不能仅凭推测实现，必须放入独立的兼容性验证阶段。

# 六、云端提供商适配器

程序内部定义统一接口：

```python
class CloudProvider:
    def authenticate(self): ...
    def get_current_user(self): ...
    def list_repositories(self): ...
    def create_private_repository(self): ...
    def get_repository(self): ...

    def read_sync_manifest(self): ...
    def write_sync_manifest(self): ...

    def list_releases(self): ...
    def create_release(self): ...
    def upload_release_asset(self): ...
    def download_release_asset(self): ...
    def delete_release(self): ...

    def acquire_lock(self): ...
    def release_lock(self): ...
    def test_connection(self): ...
    def get_capabilities(self): ...
```

分别实现：

```python
GitLinkProvider
GitHubProvider
LocalFolderProvider
```

其中`LocalFolderProvider`很有必要。

它可以把数据包同步到：

- U 盘
- 局域网共享文件夹
- 坚果云同步目录
- OneDrive 目录
- NAS 目录

这样即使 GitLink 接口暂时不稳定，软件的迁移功能仍然可用。

## 能力探测

不同提供商不一定拥有完全相同的功能，因此要返回：

```json
{
  "private_repository": true,
  "release_assets": true,
  "atomic_file_update": true,
  "oauth": false,
  "large_file_upload": true,
  "delete_release": true,
  "max_asset_bytes": null
}
```

界面根据能力决定哪些按钮可用。

不要在业务代码里大量写：

```python
if provider == "github":
```

所有平台差异都应封装在适配器内部。

# 七、远程仓库建议结构

每位用户默认使用一个私有仓库，例如：

```text
graduate-mistake-book-data
```

仓库内容：

```text
/
├── README.md
├── .mistakebook/
│   ├── repository.json
│   ├── latest.json
│   ├── devices.json
│   ├── schema.json
│   └── migration-policy.json
├── manifests/
│   ├── 000001.json
│   ├── 000002.json
│   └── 000003.json
├── changes/
│   ├── device-a/
│   └── device-b/
└── tombstones/
    └── deleted-records.jsonl
```

## repository.json

用于确认这个仓库确实属于研错库：

```json
{
  "format": "graduate-mistake-book",
  "repository_id": "repo_f93d...",
  "format_version": 1,
  "created_by_app": "1.0.0",
  "database_id": "db_6703..."
}
```

软件连接仓库时，不能只靠仓库名判断。

必须读取`repository.json`并检查：

- 格式标识
- 数据库 ID
- 格式版本
- 加密状态
- 用户是否有写入权限

# 八、云端上传流程

用户点击“上传至云端”时：

```text
检查本地数据库
→ 关闭或冻结当前写事务
→ 创建一致性快照
→ 收集新增和变化的图片
→ 计算 SHA-256
→ 生成 manifest
→ 压缩数据包
→ 可选加密
→ 上传临时附件
→ 验证远端哈希
→ 发布 Release
→ 更新 latest.json
→ 记录本次同步结果
```

不能采用：

```text
先更新 latest.json
→ 再慢慢上传附件
```

否则上传到一半失败，另一台设备可能把不完整版本误认为最新版本。

正确顺序是：

> 先上传并校验完整数据，再更新“最新版指针”。

# 九、另一台设备的迁移流程

新设备连接云端账户后：

```text
选择数据仓库
→ 读取 repository.json
→ 读取 latest.json
→ 显示云端数据摘要
→ 用户确认下载
→ 下载数据包
→ 校验 SHA-256
→ 解密
→ 在临时目录解包
→ 验证数据库版本
→ 执行迁移
→ 导入本地
→ 注册新设备
```

显示摘要：

```text
云端数据名称：张三的考研错题库
最后备份：2026-07-21 15:15
错题数量：826
图片数量：1427
数据大小：1.36 GB
来源设备：DESKTOP-A
加密状态：已加密
```

恢复过程不能直接覆盖现有数据库。

应提供：

- 覆盖本地数据
- 合并到本地数据
- 新建为另一个错题库
- 仅查看，不立即导入

# 十、同步和迁移必须分开

这两个概念很容易被混为一谈。

## 数据迁移

特点：

- 低频
- 整包上传和下载
- 一般只有一个最新版本
- 冲突较少
- Release 非常适合

例如：

> 从旧电脑迁移到新电脑。

## 多设备同步

特点：

- 高频
- 需要上传增量
- 两台设备都可能修改
- 必须解决冲突
- 不能只靠完整 Release

例如：

> 上午在安卓手机添加图片，晚上在电脑修改标签。

因此开发阶段应该分开。

---

# 十一、第一版先做“备份与迁移”，不要立即做实时双向同步

第一版云端功能建议只实现：

- 连接 GitLink 或 GitHub
- 创建或选择私有仓库
- 上传完整备份
- 查看历史备份
- 下载备份
- 恢复备份
- 删除旧备份
- 设置保留数量
- 更换设备迁移
- 手动同步按钮

同时使用“单写入设备”规则：

> 一次只能指定一台设备为主要编辑设备。

其他设备可以：

- 只读查看
- 下载
- 导入新图片到待上传区
- 申请成为主设备

这样可以避开第一版最难的多端冲突问题。

## 第二版再实现增量同步

同步单位不是整个 SQLite 文件，而是业务记录：

- 新增题目
- 修改题目字段
- 添加或删除标签
- 新增复习记录
- 删除题目
- 添加附件

每次操作生成一条变更事件：

```json
{
  "operation_id": "op_18af...",
  "device_id": "dev_android_02",
  "timestamp": "2026-07-21T15:35:20+08:00",
  "entity_type": "problem",
  "entity_id": "problem_721...",
  "operation": "update",
  "base_revision": 14,
  "new_revision": 15,
  "changed_fields": {
    "priority": 5,
    "mastery": 2
  }
}
```

# 十二、冲突解决规则

假设电脑和安卓同时修改同一道题：

- 电脑修改了解析
- 手机修改了优先级

字段不同，可以自动合并。

但如果两端都修改了解析：

```text
电脑版本 revision 15
安卓版本 revision 15
两端都基于 revision 14
```

则生成冲突。

用户需要选择：

- 使用电脑版本
- 使用安卓版本
- 并排比较
- 手动合并
- 两个版本都保留

以下字段一般可以自动合并：

- 标签增加
- 新增复习记录
- 收藏状态
- 附件新增
- 题目关系新增

以下字段发生双向修改时应人工确认：

- 原题正文
- LaTeX 公式
- 正确答案
- 解题过程
- 章节
- 删除状态

# 十三、图片存储必须去重

图片不应按照原文件名直接保存。

使用 SHA-256 作为对象标识：

```text
assets/objects/
└── 8c/
    └── 8cf92db5f0...jpg
```

数据库记录：

```json
{
  "asset_id": "asset_723...",
  "sha256": "8cf92db5f0...",
  "mime_type": "image/jpeg",
  "size": 2384912,
  "width": 3024,
  "height": 4032
}
```

好处：

- 相同图片不会重复上传
- 可以校验文件是否损坏
- 可以进行增量同步
- 文件改名不影响引用
- Windows 与安卓路径差异不会影响数据

数据库只能保存相对对象 ID，不能保存：

```text
C:\Users\张三\Pictures\题目 1.jpg
```

这种本机绝对路径。

# 十四、隐私和加密

错题图片可能包含：

- 用户姓名
- 手写内容
- 学校信息
- 购买资料
- 课程内部讲义
- API 密钥
- 个性化笔记

因此默认应使用私有仓库。

GitLink 创建项目的公开 API 包含私有项目参数，GitHub 侧也支持私有仓库；不过“私有仓库”不等于“只有用户自己可以解密”，平台仍负责托管数据。

建议提供端到端数据包加密：

```text
压缩
→ 使用随机数据密钥加密
→ 用户口令派生主密钥
→ 加密数据密钥
→ 上传密文
```

云端仅能看到：

- 文件大小
- 上传时间
- Release 名称
- 少量非敏感元数据

无法直接看到：

- 题目内容
- 图片
- 错题分析
- 复习记录

## 恢复密钥

必须明确提醒：

> 忘记加密口令后，开发者和 GitLink、GitHub 都无法恢复数据。

可以生成一个恢复文件：

```text
研错库恢复密钥.gmkey
```

恢复文件不能自动上传到同一个云端仓库。

# 十五、安卓端技术路线修正

Windows 和安卓确实应该采用不同 UI。

但是不只是“UI 不能相同”，开发技术也不建议强求全部使用 Python。

## Windows 端

继续采用：

- Python
- PySide6
- SQLite
- SQLAlchemy
- Markdown 与 LaTeX 渲染

## 安卓端

更推荐：

- Kotlin
- Jetpack Compose
- Room 或直接使用 SQLite
- Android WorkManager 处理后台任务
- Android Keystore 保存令牌和密钥

也可以选择 Flutter，但不要为了统一语言而强行使用 Python 安卓框架。

Python 能够通过某些框架运行在安卓上，但在以下方面通常会增加大量成本：

- 原生文件选择器
- 相机和相册权限
- 后台上传
- 系统分享
- 安全密钥存储
- 应用体积
- UI 适配
- 系统生命周期
- 应用商店发布

## 真正需要共享的不是 UI 代码

两个客户端应共享：

- 数据格式
- 数据库迁移规则
- JSON Schema
- 云端同步协议
- `.ebpack`格式
- 加密格式
- 哈希算法
- Release 命名规则
- 冲突规则
- API 请求约定

因此可以建立一个独立规范目录：

```text
protocol/
├── data-format-v1.md
├── sync-protocol-v1.md
├── encryption-v1.md
├── schemas/
│   ├── problem.schema.json
│   ├── manifest.schema.json
│   └── operation.schema.json
└── test-vectors/
```

Windows 和安卓分别按照同一个规范实现。

# 十六、安卓端的功能应当有所取舍

安卓端第一版最适合的定位是：

> 快速采集、简单整理、移动复习。

建议包括：

- 拍摄错题
- 从相册选择图片
- 批量上传到收件箱
- 查看错题
- 搜索筛选
- 修改标签
- 调整优先级
- 记录复习结果
- 查看公式和解析
- 下载离线数据
- 手动同步

第一版不急于实现：

- 复杂 Word 排版
- 高级 PDF 编辑
- 外部工作区同步
- 大规模 AI 批量编辑
- 复杂模板设计器
- 完整数据库管理
- 插件系统

这些仍然放在 Windows 端完成。

# 十七、朋友之间的数据分享

个人云同步和朋友分享是两套功能，不能混在一起。

## 个人同步

每个人一个私有仓库：

```text
张三/graduate-mistake-book-data
李四/graduate-mistake-book-data
```

## 朋友分享

可以导出一个不包含私人复习记录的分享包：

```text
高数不定积分精选错题.gmshare
```

用户可以选择是否包含：

- 原题
- 正确答案
- 解析
- 标签
- 来源
- 原始图片
- 用户错误过程
- 个人备注
- 复习记录

默认不分享：

- 用户错误手写
- 私人备注
- 复习历史
- AI 密钥
- 云端配置
- 设备信息

导入朋友分享包后，题目应获得新的本地 ID，同时保留：

```json
{
  "origin_package_id": "...",
  "origin_problem_id": "...",
  "imported_from": "shared-package"
}
```

避免再次导入时重复创建。

# 二十、最终推荐方案

整个系统可以概括为：

```text
Windows Python 客户端
        │
        ├── 本地 SQLite 数据库
        ├── 本地图片对象库
        └── 云端适配器
                ├── GitLink
                ├── GitHub
                └── 本地文件夹
                         │
                         ▼
                私有远程仓库
                ├── 同步清单
                ├── 设备信息
                ├── 增量操作
                └── Release 完整备份
                         ▲
                         │
Android Kotlin 客户端
```

最重要的四条结论是：

1. GitLink 优先、GitHub 备用是可行的，但必须通过统一适配器切换。
2. Release 用于完整备份和迁移，不用于每次编辑的实时同步。
3. Release 标签只标识版本，真正的数据描述放进 manifest。
4. Windows 和安卓共享数据协议，不必强求共享 Python 代码或 UI。
